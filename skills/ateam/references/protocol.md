# ateam Sentinel Protocol Reference

## Overview

Terminal-based teammates (Codex CLI, Gemini CLI) communicate structured responses using sentinel markers embedded in their terminal output. The orchestrator sends tasks with a sentinel ID, and the teammate wraps their response in matching markers. This protocol does NOT apply to Claude teammates, which use native `SendMessage`.

## Marker Formats

### Response (task completed)

```
TEAM_RESPONSE_START <sentinel-id>
[response content — summary of work done, files changed, decisions made]
TEAM_RESPONSE_END <sentinel-id>
```

### Blocked (teammate needs help)

```
TEAM_BLOCKED_START <sentinel-id>
[description of what the teammate needs to proceed]
TEAM_BLOCKED_END <sentinel-id>
```

### Idle (nothing to do)

```
TEAM_IDLE <sentinel-id>
```

## Sentinel ID Format

Each task message carries a unique sentinel ID generated by the orchestrator:

```
TEAM_MSG_{messageCount+1}
```

Where `messageCount` is the current count for that specific teammate (tracked in `team.json`). The teammate must use the sentinel ID from the most recent task message they received.

## Sending Tasks to Terminal Teammates

1. Generate sentinel ID: `TEAM_MSG_{messageCount+1}`
2. Build prompt from template (`prompts/task-assignment.md` or `prompts/status-check.md`)
3. Replace `{{SENTINEL_ID}}` with the generated ID
4. Replace `{{TASK_DESCRIPTION}}` with the actual task
5. Write completed prompt to temp file: `/tmp/ateam-{team}-{name}-msg{N}.txt`
6. Send via: `python3 <skill-path>/fork-terminal/tools/send_to_surface.py --surface <ref> --file <path>`

## Reading Responses from Terminal Teammates

```bash
python3 <skill-path>/ateam/tools/read_teammate.py \
  --surface <ref> \
  --start-marker TEAM_RESPONSE_START \
  --end-marker TEAM_RESPONSE_END \
  --blocked-start TEAM_BLOCKED_START \
  --blocked-end TEAM_BLOCKED_END \
  --idle-marker TEAM_IDLE \
  --sentinel-id TEAM_MSG_N
```

Add `--poll --interval 5 --max-attempts 120` to wait for the teammate to finish.

## Return Values

The `read_teammate.py` tool returns JSON with a `status` field:

| Status | Meaning | Content |
|--------|---------|---------|
| `response_found` | Teammate completed the task | `content` has the response text |
| `blocked` | Teammate needs help | `content` describes what they need |
| `idle` | Teammate has nothing to do | No content |
| `no_sentinel` | No markers found yet | `lastLines` has raw terminal output |
| `timeout` | Poll exhausted max attempts | `attempts` count + `lastLines` |
| `error` | Failed to read terminal | `message` with error details |

## ANSI Handling

Terminal output contains ANSI escape codes (colors, cursor movement). The `read_teammate.py` tool strips these automatically before parsing. Teammates should output markers as plain text — ANSI formatting around the markers is tolerated.

## Bootstrap

When a terminal teammate is first spawned, the orchestrator sends a bootstrap prompt (from `prompts/teammate-bootstrap.md`) via `fork_terminal.py --delayed-input-file`. This prompt:

1. Introduces the teammate's name and role
2. Explains the sentinel protocol
3. Includes the first task with its sentinel ID

The teammate uses this bootstrap to understand the communication protocol for all subsequent messages.
